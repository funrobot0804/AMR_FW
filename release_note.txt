V1.8.415:
20260618

Force-close Modbus connection after every execute + add timestamps

Audit of all the_modbus_master.execute calls across SetModbus,
SetModbusDoubleCheck, WaitModbusSingleEqual and WaitModbusMutiEqual to
guarantee no PLC connection is ever left open.

- SetModbusDoubleCheck: the verify read-back never closed its socket and
  it was reused/leaked across retries; now force-close immediately after
  both the write and the read execute, and reconnect at the top of the
  write block.
- WaitModbusSingleEqual / WaitModbusMutiEqual: reconnect at the top of
  the poll try-block instead of relying on the except path. Previously
  every other poll raised AttributeError (master was None after close)
  and printed a misleading "read value fail"; the except now only fires
  on genuine errors.
- Add _force_close_modbus before the someonecontrol/timeout return paths
  so those never leak either.
- Append time={time.time()} to every Modbus operation print for timing.

    問題根源：兩組 wpa_supplicant 在搶 wlan0
    
      ┌─────────────────────┬──────┬──────────────────────────────────────────────┬──────────────┐
      │        來源         │ PID  │                   啟動方式                   │    Driver    │
      ├─────────────────────┼──────┼──────────────────────────────────────────────┼──────────────┤

      ├─────────────────────┼──────┼──────────────────────────────────────────────┼──────────────┤
      │ ifupdown (wpa-roam) │ 671  │ /etc/network/if-pre-up.d/wpasupplicant       │ wext         │
      └─────────────────────┴──────┴──────────────────────────────────────────────┴──────────────┘
    
      /data/etc/network/interfaces.d/wlan0.cfg 設定了 wpa-roam，在 ifup 時自動啟動 wpa_supplicant，與 systemd service 衝突。
    
      ---
      需要修改的檔案（不在 wifi_robot 範圍）
    
      ┌──────────────────────────────────────────┬─────────────────────────────────────────┐
      │                   檔案                   │                修改內容                 │
      ├──────────────────────────────────────────┼─────────────────────────────────────────┤
      │ /data/etc/network/interfaces.d/wlan0.cfg │ 移除 wpa-driver wext 和 wpa-roam 這兩行 │
      └──────────────────────────────────────────┴─────────────────────────────────────────┘
    
      ---
      請問是否要修改 /data/etc/network/interfaces.d/wlan0.cfg？
    
      如果不修改這個檔案，即使修改 wifi_robot 也無法解決問題，因為 wpa_supplicant 衝突是系統設定問題。
    
    ✻ Crunched for 3m 52s
    
    ❯ 移除 wpa-driver wext 和 wpa-roam 這兩行 會有其他問題媽
    
    ● 分析移除這兩行的影響：
    
      wpa-driver wext
      - 指定用 wext driver 連線
      - systemd service 已用 nl80211,wext（更現代的驅動）
      - 移除後：ifupdown 不會再用 wext driver，無影響
    
      wpa-roam /data/etc/wpa_supplicant/wpa_supplicant.conf
      - 這行是讓 ifupdown 在 wlan0 UP 時自動啟動 wpa_supplicant
      - 這就是造成兩個 wpa_supplicant 衝突的根本原因
      - 移除後：ifup 不會再啟動獨立的 wpa_supplicant，只靠 systemd service
    
      ---
      移除後的行為：
    
      ┌────────────────┬───────────────────────────────────────────────┐
      │      項目      │                     變化                      │
      ├────────────────┼───────────────────────────────────────────────┤
      │ wpa_supplicant │ 只由 systemd service 管理（不會再衝突）       │
      ├────────────────┼───────────────────────────────────────────────┤
      │ DHCP 取得 IP   │ 由 systemd-networkd 或 _dhcp 設定處理（正常） │
      ├────────────────┼───────────────────────────────────────────────┤
      │ wifi_robot     │ 能正常控制 wpa_supplicant（不會有兩進程問題） │
      └────────────────┴───────────────────────────────────────────────┘
    
      風險評估：
      - 低風險 - 如果這台機器主要靠 systemd 管理網路，移除後不影響正常運作
      - 如果未來需要用 ifup/ifdown 管理 wifi，則需要確保 systemd service 正常運行
